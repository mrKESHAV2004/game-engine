§ 1.1

What is a Game Engine?

At its core, a game engine is a strict hardware abstraction layer. Its single job is to isolate pure game logic from the volatile, platform-specific demands of the Operating System (OS) and Graphics Processing Unit (GPU).

Instead of writing code that talks directly to Windows, macOS, or a specific graphics card, game developers write "Game Code" — like making a character jump. The engine translates that jump into the complex math and hardware instructions needed to update physics and render pixels. It removes magic and provides a predictable, data-oriented environment running in a tightly controlled loop.

Key insight: The engine is not the game. The engine is the operating system your game runs on top of. Every decision you make building the engine will constrain — or liberate — the game teams that use it for years.

The Mental Model

🎮

Game Code

The logic unique to your game: character stats, level design, dialogue. This should be 100% portable — it never calls OpenGL or reads raw keyboard state directly.

⚙️

Engine Layer

The reusable machinery: renderer, physics, audio, ECS, asset manager. Platform-agnostic. The engine translates game intent into hardware commands.

🖥️

OS / Hardware

The actual machine: GPU drivers, OS windowing, file system, audio device. The engine isolates all game code from this volatile, vendor-specific layer.

§ 1.2

Core Components

An engine is broken into specific subsystems, each handling a different hardware responsibility. Every subsystem is a contained module with a well-defined interface to the rest of the engine.

🖼️

Renderer Pipeline

Collects draw calls from ECS systems, sorts by layer and texture, batches sprites sharing an atlas into single GPU draw calls, executes shaders, swaps framebuffer. Sprite batching is the single highest-leverage optimisation in 2D.

OpenGL Vulkan Texture Atlas Z-sorting
🔩

ECS — Entity-Component-System

Entities are numeric IDs. Components are plain data structs stored in contiguous arrays. Systems iterate matching arrays — no virtual dispatch, no OOP inheritance, cache-friendly by construction.

EnTT flecs Sparse-set Archetype
🧲

Physics Engine

Broadphase (spatial hash) quickly rules out impossible collisions. Narrowphase (SAT/GJK) tests candidate pairs precisely. Impulse resolution applies corrective forces. Trigger sensors fire events without blocking movement.

Box2D rapier2d SAT GJK
📦

Asset Manager

Handle-based loader: request returns an opaque handle immediately; loading is async. A cache map prevents duplicate decodes. Reference counting drives eviction — assets unload when no handle remains.

stb_image Hot reload Ref-count
🎮

Input System

OS events polled once per real frame into a platform-agnostic state map. Action mapping layer binds logical actions (Jump, Fire) to physical buttons — enabling remapping without touching game code.

SDL3 Action map Input buffer
🔊

Audio Engine

Sound emitters pass through spatial processing (distance attenuation, panning). SFX, music, and voice hit separate mixer channels. Both paths merge at master mixer before the audio device.

miniaudio OpenAL Positional
📡

Event Bus

Typed event structs. Publishers emit; subscribers register by event type. Dispatch deferred to end-of-tick to prevent re-entrant mutation during system iteration. Zero coupling between producers and consumers.

Pub/sub Deferred Typed events
🗺️

Scene Manager

Named scene registry with async loading. Stacked scenes let a pause menu overlay gameplay without unloading it. Transition hooks (fade, cut) allow smooth scene changes. OnEnter/OnExit lifecycle events per scene.

Stack Async load Transitions
§ 1.3

How to Start

The golden rule: Build the loop first. Draw a sprite. Ship something playable before writing a single line of editor code. Do not start with the ECS or a physics engine.

The Skeleton rule: The first milestone is exactly this — a window opens, a sprite appears, WASD moves it, the window does not crash. Everything else is Phase 2 or later.

§ 1.4

Testing Each System

Early-stage engine testing is visual. You rely on console logging, colour-coded overlays, and failure modes that are unmistakable.

🪟

Testing the Window & Platform Layer

Open the window. Change the clear colour from code (glClearColor(r, g, b, 1)). If the window opens without crash and the colour changes, the platform layer is functional. Try resizing — does the viewport update? Try closing — does it exit cleanly?

⏱️

Testing the Game Loop

Move your single sprite using pos.x += vel * fixed_dt. Then intentionally call sleep(1000ms) for one second. If the game crashes trying to execute 60 backed-up physics ticks, your loop is broken — add the 250ms cap. Speed should be identical at 30fps and 144fps.

⌨️

Testing Input

Map WASD to sprite XY movement. Verify IsPressed() only fires on the first frame of a keypress. Verify IsReleased() fires exactly once. Log input events to console for the first week — you'll catch subtle bugs that are impossible to see otherwise.

🎨

Testing the Renderer / Sprite Batching

Open RenderDoc and capture a frame. Count the draw calls. If you have 200 sprites using the same atlas and see 200 draw calls, batching is broken. Correctly batched: exactly 1 draw call per unique atlas. This is your renderer's primary correctness test.

🧱

Testing Physics & Collisions

Write a debug renderer that draws wireframe AABB boxes over every collider. Boxes turn red on intersection, green when clear. Fire OnCollision and log to console. This visual test will save you weeks of debugging invisible collision bugs.

📦

Testing the Asset Manager

Request the same texture handle twice. Verify the file is opened only once. Drop one handle — asset stays loaded. Drop both — asset evicts. Run with AddressSanitizer — any double-free in your ref-counting will surface immediately.

§ 2.1

The 3 Ironclad Rules

These rules are not suggestions. Every system in your engine must satisfy all three. Violating any one will eventually force a full architectural rewrite.

01

Strict Isolation

Layers only speak to the layer directly below. Game code never calls the GPU or reads raw hardware input. The renderer never reads from the ECS directly.

02

Explicit Over Implicit

No magic. If a system runs, it must be visible in the scheduler. Implicit dependencies cause frame-order bugs that only appear on specific hardware at specific frame rates.

03

Data-Oriented Design

Components are plain data structs — no virtual methods, no inheritance. Systems iterate contiguous arrays. A CPU cache line is 64 bytes: keep it hot.

§ 2.2

5-Tier Architecture

Dependencies flow strictly downwards. A module may call into the layer below it. It may never call into the layer above it — only emit events through the event bus that upper layers subscribe to.

Enforcement mechanisms

  • →CMake library targets — Each layer is a separate CMake target. Cross-layer includes that violate the dependency direction fail to link. No compiler warnings: build failures.
  • →Forward declarations — Never include a higher-layer header from a lower-layer module. Use forward declarations at every boundary.
  • →Event bus as upward channel — The only way a lower layer communicates upward is by emitting a typed event. Upper layers subscribe. No direct calls upward ever.
  • →Dependency graph review — Run cmake --graphviz=deps.dot quarterly. Flag any cycles immediately.
§ 2.3

The Fixed-Timestep Game Loop

The game loop is the heartbeat. A deterministic simulation requires a fixed timestep, while rendering runs at a variable rate. The accumulator pattern solves this by separating the two.

✓ Fixed-rate update path

Physics and game logic run exactly at FIXED_DT = 1/60s. This guarantees determinism: the same input always produces the same output. Essential for replays, networking, and physics stability.

✓ Variable-rate render path

Rendering runs as fast as the GPU allows. The alpha factor (remainder / fixed_dt) linearly interpolates object positions between physics ticks, eliminating visual jitter on high-refresh monitors.

System execution order within each fixed tick

Step System Why this order
1Input pollMust be first — all downstream systems read from this frame's input snapshot
2Physics stepPhysics must see this frame's input before game logic does
3Game logic / ECSReads physics results; applies game rules, AI, cooldowns
4Animation updateReads final transform; advances sprite frames, fires timeline events
5Event dispatchFlushed after all systems — prevents re-entrant mutation during iteration
6Camera updateFollows targets after they've moved; camera lag needs final positions
7Render (free-rate)Outside the fixed loop — uses interpolated positions, submits to GPU

Common pitfalls

Spiral of Death

If each physics tick takes longer than FIXED_DT, the accumulator grows faster than it drains. Fix: cap frame_time = min(frame_time, 0.25).

Missing dt Multiply

Any velocity applied without multiplying by dt becomes frame-rate dependent. At 120fps the game runs 2x faster than at 60fps.

Rendering Physics Positions Directly

Without interpolation, objects jitter visibly when render rate differs from physics rate. Always: render_pos = prev + (curr - prev) * alpha.

Reading Input Inside Systems

If two physics ticks fire in one real frame, they must see the same input state. Poll input exactly once per real frame, before the fixed loop begins.

§ 2.5

16.67ms Frame Budget

Every architectural rule exists to ensure all game logic completes within the 60 FPS frame boundary. At 60Hz, you have exactly 16.67 milliseconds. This is your budget. There is no overdraft.

Physics ~5ms
Logic ~3ms
Render ~6ms
Misc ~2.7ms
Physics / collision
Logic / ECS systems
Render / GPU submit

Budget by target frame rate

30 fps33.3 ms~10 ms~5 ms~12 ms~4 ms

Profiling rule: Do not guess where time is spent. Use Tracy or Remotery to instrument each system with named zones. The bottleneck is almost never where you expect it. Measure first, optimise second.

Part III
Team Concerns — Deep Dives

Five concerns raised during planning. Each is analysed with its root risk, concrete mitigations, and code-level guidance.

⚠ 4.1

Graphics API — Selection & Lock-in

Team concern — verbatim

"Which graphics API do we commit to, and are we locking ourselves in if we pick the wrong one early?"

Root risk

Choosing Vulkan too early adds months of driver, synchronisation, and memory-management work before a single sprite appears on screen. Choosing OpenGL legacy limits future platform options (macOS caps at 4.1). Choosing a high-level abstraction (bgfx) hides important concepts from the team.

The RHI solution

Design a thin Rendering Hardware Interface (RHI) — a pure-virtual abstraction that all rendering code calls. Concrete implementations sit behind it. The two-day cost of building the RHI in Phase 1 buys a backend swap at any time for free.

C++ — interface Minimal RHI abstraction
class IRHI { public: virtual TextureHandle CreateTexture(const TextureDesc& desc) = 0; virtual BufferHandle CreateBuffer(const BufferDesc& desc) = 0; virtual void SetShader(ShaderHandle shader) = 0; virtual void DrawIndexed(uint32_t count) = 0; virtual void Present() = 0; }; // Game code never sees concrete backends: class OpenGLRHI : public IRHI { /* ... */ }; class VulkanRHI : public IRHI { /* swap in at Week 19 if needed */ };
  • →OpenGL 4.6 core profile to start. Well-documented, runs on all desktop hardware, great tooling (RenderDoc). No legacy cruft from core profile.
  • →RHI interface from day one. ~2 days of overhead in Phase 1. Eliminates the cost of a future backend swap entirely.
  • →Formal Vulkan decision gate at Week 19. Run RenderDoc and Tracy. Is the OpenGL submission overhead actually a bottleneck under your target load? Evidence-based, not preference-based.
  • →macOS consideration. macOS supports OpenGL only to 4.1. Validate all shaders at 4.1 feature level from the start in CI. Use Metal via MoltenVK if 4.6 features are needed.
⚠ 4.2

Multithreading — Model & Safety

Team concern — verbatim

"We don't want to build single-threaded and have to retrofit threading later. But we also don't want threading bugs to block early iteration."

Threading introduction schedule

Thread Introduced Owns Communicates via
Main / LogicPhase 1ECS update, input, AI, physics, sceneSingle-threaded — direct calls
Render threadPhase 2All GPU state, all RHI calls, framebufferDouble-buffered RenderCommandBuffer
Job poolPhase 3Parallel ECS systems, pathfinding, AIComponent access declarations → scheduler
Streaming threadPhase 4Async asset decode, texture stagingLock-free request queue → completion poll
  • →The GPU thread owns all GPU resources. No other thread may call RHI functions. Enforce this with thread-ID assertions in debug builds — violations become immediate crashes, not silent corruption.
  • →ThreadSanitizer (TSan) in CI from Phase 2. Run the full test suite under TSan on every PR. Data races surface immediately. Do not defer — a race found in Phase 4 costs 10× more to fix than one found in Phase 2.
  • →Lock-free queue for streaming. The streaming thread and main thread communicate via a lock-free MPSC queue. The main thread polls for completions; it never blocks waiting for an asset.
⚠ 4.3

Layering — Enforcing Boundaries

Team concern — verbatim

"How do we prevent every system from depending on every other system? We've seen this cause spaghetti in past projects."

What spaghetti looks like in practice

Renderer reads from ECS

The renderer directly queries entity components instead of reading a command buffer. Now Renderer depends on ECS — a Core system depends on an ECS concept. The layer cake collapses.

Physics spawns entities

The physics resolution code creates a spark entity when objects collide. Now Physics depends on the ECS entity factory. Two Core systems are coupled. Neither can be tested in isolation.

Game code calls OpenGL

A script calls glBlendFunc directly. Now game code depends on the GPU API. Porting to Vulkan or WASM requires touching every script that does this.

  • →Separate CMake library targets per layer. Platform, Core, ECS, Game are four separate CMake static libraries. Cross-layer violations fail at link time — not review time.
  • →Event bus as the only upward channel. When Physics needs to tell Game that a collision occurred, it emits a CollisionEvent to the bus. Game code subscribes. Physics never holds a pointer to any game object.
  • →Quarterly graphviz review. Run cmake --graphviz=deps.dot build/ and render it. If any arrow points upward through the layer hierarchy, that is an architectural bug — fix it before it multiplies.
⚠ 4.4

Garbage Collectors — Pauses & Memory

Team concern — verbatim

"If we bind a scripting language with a GC (Lua, Python), will GC pauses kill our frame timing? And how do we handle C++ memory without leaks?"

Scripting layer GC risk

Lua 5.4 — tunable incremental GC

Lua's GC is incremental and tunable. The risk is real if scripts allocate per-frame (creating tables, closures, string concatenations inside update()).

Low risk if rules followed

Python — reference counter + cyclic GC

CPython's cyclic GC can pause. Python should only be used for editor tooling, not the shipped game loop — the GC issue is irrelevant at runtime.

Runtime: avoid. Tooling: fine.
C++ / Lua Tuning Lua GC for real-time use
// Tune Lua GC at engine startup: lua_gc(L, LUA_GCSETSTEPMUL, 100); // step size (lower = smaller pauses) lua_gc(L, LUA_GCSETPAUSE, 200); // threshold (higher = rarer) // Force a full collect ONLY during scene transitions: void OnSceneUnload() { lua_gc(L, LUA_GCCOLLECT, 0); // safe — screen is black anyway } // THE RULE: No Lua allocations inside update(). // All tables/closures created in init() and reused.

C++ engine memory strategy

  • →No raw new/delete in engine code. Use std::unique_ptr at allocation sites and std::shared_ptr only at actual shared-ownership boundaries.
  • →Frame allocator for hot path. Allocate a 16 MB arena at startup. Each frame: a bump pointer starts at the arena base. Allocations just increment the pointer. At end-of-frame: reset. Zero fragmentation, zero individual frees.
  • →ASan + UBSan in CI from day one. Every PR runs with AddressSanitizer and UndefinedBehaviourSanitizer. Double-frees, use-after-free, and buffer overflows surface as immediate test failures.
  • →Heaptrack audit at Week 22. Run Heaptrack under full game load. Confirm: zero heap allocations per frame on the main thread. All hot-path memory comes from arenas or pre-allocated pools.
⚠ 4.5

Shaders — Pipeline & Team Access

Team concern — verbatim

"Shaders are a black box for half the team. How do we structure the shader pipeline so engine and graphics programmers don't block each other?"

Phased shader system — no bottleneck

Phase Feature Owner What it unlocks
1Hard-coded sprite shaderGraphics leadSprite on screen. No abstraction — keep it simple.
2ShaderLibraryEngine teamNamed map of compiled programs. Hot-reload on file change in debug builds.
3Material systemEngine teamMaterial = (ShaderHandle, UniformMap). Game code creates materials; engine manages GPU state.
4Permutation systemGraphics teamVariants via #define injection at load. NO_LIGHTING, POINT_LIGHT, NORMAL_MAP — no runtime branching.
4Post-processing pipelineGraphics teamComposable passes: bloom → chromatic aberration → colour grade. Each pass reads/writes RenderTarget.
GLSL + C++ Shader permutation via define injection
struct ShaderKey { bool lighting = false; bool normal_map = false; bool outline = false; }; std::string build_preamble(ShaderKey key) { std::string p = "#version 460 core\n"; if (key.lighting) p += "#define USE_LIGHTING\n"; if (key.normal_map) p += "#define USE_NORMAL_MAP\n"; if (key.outline) p += "#define USE_OUTLINE\n"; return p; } // shader.glsl uses #ifdef — one file, N variants, no runtime branching
  • →ShaderLibrary owned by the render thread. The main thread enqueues a hot-reload request via the command buffer. The render thread picks it up between frames and recompiles. Game code is never blocked.
  • →Offline compilation in release builds. Cache compiled shader binaries on disk. Binary keyed by source hash — stale cache auto-invalidates. Eliminates per-launch compilation stall for end users.
  • →Variant explosion guard. Document the maximum variant count per shader (e.g. max 8 permutations of the sprite shader). Beyond 8, the compiled program count grows exponentially. Set the limit in Phase 4 before the shader library gets large.

§ 5.2

Glossary

Term Definition
ADRArchitecture Decision Record — a short document recording a significant technical decision and its rationale.
alphaThe interpolation factor (accumulator / fixed_dt) passed to the renderer to smooth object positions between physics ticks.
ASanAddressSanitizer — a compiler instrumentation tool that detects memory errors (use-after-free, buffer overrun) at runtime.
ECSEntity-Component-System — entities are IDs, components are data, systems contain all logic. No OOP inheritance.
Frame allocatorA linear bump allocator reset at the end of each frame. Zero fragmentation, zero individual frees on the hot path.
RHIRendering Hardware Interface — a thin abstraction over a specific GPU API (OpenGL, Vulkan). Game code never calls the GPU directly.
SATSeparating Axis Theorem — a narrowphase algorithm that tests convex polygon intersection by checking if any separating axis exists.
Shader permutationA compiled variant of a shader produced by injecting #define flags at load time. Avoids runtime branching in GLSL.
Spiral of deathThe fixed-timestep loop failure mode where each physics tick takes longer than fixed_dt, causing the accumulator to grow infinitely.
TSanThreadSanitizer — detects data races at runtime. Run in CI from the moment multithreading is introduced.
UBSanUndefinedBehaviourSanitizer — detects C++ undefined behaviour (signed overflow, null deref, misaligned access) at runtime.
Z-sortingSorting draw calls by depth layer and Y-position within a layer to determine render order of overlapping sprites.
